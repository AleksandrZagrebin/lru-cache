# LRU-cache

**Cache** — это временное высокоскоростное хранилище, которое ускоряет доступ к данным за счёт сохранения копий часто используемой информации. Он работает на основе принципа локальности: если данные запросили один раз, вероятно, они понадобятся снова. Кеши используются повсеместно — в процессорах, базах данных, веб-браузерах, CDN-сетях и операционных системах. Без кеширования современные системы работали бы на порядок медленнее, но из-за ограниченного размера возникает задача эффективного управления памятью.


## LRU-кеш

**LRU-кеш** (Least Recently Used) — это алгоритм управления кешем, который при заполнении удаляет элемент, к которому дольше всего не обращались. Он реализуется с помощью хеш-таблицы для быстрого поиска за O(1) и двусвязного списка для быстрой вставки и удаления за O(1) с отслеживанием порядка использования: голова — самый свежий элемент, хвост — самый старый. При каждом обращении элемент перемещается в голову, а при переполнении удаляется хвост, что обеспечивает постоянную скорость выполнения всех операций независимо от размера кеша. LRU широко применяется в веб-браузерах, базах данных и системах кеширования благодаря простоте и эффективности в сценариях с локальностью обращений.

## Постановка задачи

**Тема проекта:** Разработка LRU-кеша на языке C с использованием хеш-таблицы и двусвязного списка.

**Актуальность:** Кеширование данных является ключевым механизмом повышения производительности современных компьютерных систем. LRU-кеш (Least Recently Used) — один из наиболее распространённых алгоритмов управления кешем, используемый в процессорах, базах данных, веб-браузерах, CDN-сетях и операционных системах. Эффективная реализация LRU-кеша требует грамотного управления памятью и оптимизации операций поиска, вставки и удаления.

**Цель работы:** Реализовать эффективный LRU-кеш на языке C, обеспечивающий выполнение операций поиска, вставки и удаления за O(1), с полным управлением памятью и покрытием юнит-тестами.

**Задачи проекта:**

1. Разработать модуль хеш-таблицы с цепочечным разрешением коллизий, автоматическим рехешированием при превышении коэффициента загрузки 0.75 и файловой инкапсуляцией через opaque pointer. Ключи хеш-таблицы — целочисленные, значения — указатели на void для хранения данных любого типа.

2. Разработать модуль двусвязного списка для поддержания порядка использования элементов. Список организован таким образом, что голова содержит самый свежий элемент, а хвост — самый старый. Список должен обеспечивать операции добавления в голову, удаления из хвоста, перемещения узла в голову и удаления произвольного узла.

3. Реализовать LRU-кеш, объединяющий хеш-таблицу и двусвязный список. Кеш должен предоставлять методы создания с заданной ёмкостью, вставки и обновления элементов, получения элементов с перемещением в голову, автоматического вытеснения самого старого элемента при переполнении и подсчёта количества попаданий (Cache_hit). При каждом успешном обращении к существующему элементу счётчик Cache_hit увеличивается.

4. Обеспечить инкапсуляцию каждого модуля через opaque pointer, скрывая внутренние структуры и поля от пользователя. Пользователь должен работать только через публичный интерфейс, что гарантирует целостность данных и упрощает поддержку кода.

5. Разработать набор юнит-тестов для каждого модуля, покрывающих все сценарии использования: создание и уничтожение, вставку и поиск, обновление, удаление, коллизии, рехеширование, работу с отрицательными ключами, хранение строковых значений и проверку корректности счётчика Cache_hit.

6. Проверить отсутствие утечек памяти через Valgrind с флагами --leak-check=full и --show-leak-kinds=all для каждого модуля и основного приложения.

**Функциональные требования:**

1. Хеш-таблица должна автоматически выполнять рехеширование при превышении коэффициента загрузки 0.75, увеличивая размер вдвое с округлением до простого числа.

2. Двусвязный список должен обеспечивать выполнение всех операций за O(1) и позволять хранить данные любого типа через указатель на void.

3. LRU-кеш должен при вставке нового элемента помещать его в голову списка и добавлять в хеш-таблицу. При обращении к существующему элементу он должен перемещаться в голову списка, а счётчик Cache_hit — увеличиваться. При переполнении должен удаляться самый старый элемент из хвоста списка и соответствующая запись из хеш-таблицы.

4. Управление памятью: пользователь самостоятельно выделяет и освобождает память для данных. Кеш, хеш-таблица и список освобождают только свои внутренние структуры, не трогая пользовательские данные.


## Хеш-таблица

В рамках проекта разработана собственная хеш-таблица, реализующая метод цепочек для разрешения коллизий. Это означает, что каждый элемент массива (корзина) является указателем на связный список узлов, что позволяет хранить несколько ключей с одинаковым хешем. Ключи хеш-таблицы являются целочисленными, а значения хранятся в виде указателей на тип void, что позволяет использовать её для хранения данных любого типа. Размер таблицы всегда округляется до простого числа для уменьшения вероятности коллизий. Хеш-функция вычисляется как остаток от деления абсолютного значения ключа на размер таблицы, что обеспечивает равномерное распределение и высокую скорость вычислений.

Для работы с хеш-таблицей реализованы следующие методы:

- `hash_table_create()` — создание таблицы с размером, округлённым до простого числа;
- `hash_table_put()` — вставка или обновление значения по ключу с автоматическим рехешированием при превышении коэффициента загрузки 0.75;
- `hash_table_get()` — поиск значения по ключу за O(1);
- `hash_table_remove()` — удаление элемента по ключу;
- `hash_table_clear()` — очистка всех элементов;
- `hash_table_free()` — полное освобождение памяти;
- `hash_table_count()` — получение количества элементов;
- `hash_table_size()` — получение текущего размера таблицы;
- `hash_table_print()` — отладочный вывод содержимого.

Хеш-таблица спроектирована как самостоятельный модуль с файловой инкапсуляцией: в заголовочном файле объявлен только непрозрачный указатель на структуру `struct HashTable_t`, а все внутренние структуры (`HashNode_t`, поля `buckets`, `size`, `count`, `load_factor`) и вспомогательные функции скрыты в файле реализации. Это обеспечивает полную изоляцию внутренней реализации и защиту данных от прямого доступа.

Управление памятью в хеш-таблице построено на принципе, что пользователь самостоятельно выделяет и освобождает память для значений. Хеш-таблица освобождает только свои внутренние узлы, не трогая данные, что даёт пользователю полный контроль над временем жизни объектов и позволяет использовать хеш-таблицу в различных сценариях без риска двойного освобождения памяти.


## Двусвязный список

В рамках проекта разработан двусвязный список, который используется для поддержания порядка использования элементов в LRU-кеше. Список организован таким образом, что голова всегда содержит самый свежий (недавно использованный) элемент, а хвост — самый старый. Это позволяет эффективно перемещать элементы при каждом обращении и быстро удалять наименее используемый элемент при переполнении кеша. Значения в списке хранятся в виде указателей на тип void, что позволяет хранить данные любого типа.

В списке реализованы следующие методы:

- `list_create()` — создание пустого списка;
- `list_free()` — полное освобождение памяти;
- `list_push_front()` — добавление элемента в голову;
- `list_pop_back()` — удаление элемента из хвоста;
- `list_move_to_front()` — перемещение существующего элемента в голову (используется при cache hit);
- `list_remove_node()` — удаление произвольного элемента;
- `list_clear()` — очистка всех элементов;
- `list_head()` — получение головы списка;
- `list_tail()` — получение хвоста списка;
- `list_size()` — получение количества элементов;
- `list_empty()` — проверка, пуст ли список;
- `list_print()` — отладочный вывод содержимого.

В проекте реализована **полная инкапсуляция** двусвязного списка. Структура самого списка (`struct List_t`) полностью скрыта от пользователя: в заголовочном файле объявлен только непрозрачный указатель, а все внутренние поля  и вспомогательные функции находятся в файле реализации. Это защищает внутреннее состояние списка от случайных изменений и гарантирует, что пользователь работает только через публичный интерфейс.

Структура узла (`struct ListNode_t`) также полностью скрыта от пользователя. Для доступа к полям узла реализованы геттеры и сеттеры. Это позволяет LRU-кешу работать с узлами списка через единообразный интерфейс, не нарушая инкапсуляцию и сохраняя контроль над доступом к данным. 

Таким образом, обеспечивается полная изоляция внутренней реализации списка, упрощается поддержка кода и модификация внутренней логики без изменения пользовательского кода, что делает модуль гибким и надёжным.

## LRU-кеш

В рамках проекта разработан LRU-кеш (Least Recently Used), который объединяет хеш-таблицу и двусвязный список для эффективного управления ограниченным объёмом памяти. Кеш состоит из трёх основных компонентов: хеш-таблицы для быстрого поиска элементов по ключу за O(1), двусвязного списка для поддержания порядка использования (голова — самые свежие элементы, хвост — самые старые), а также счётчика Cache_hit, который отслеживает общее количество попаданий в кеш. Значения в кеше хранятся в виде указателей на тип void, что позволяет хранить данные любого типа.

Для работы с LRU-кешем реализованы следующие методы:

- `lru_create()` — создание кеша с заданной ёмкостью;
- `lru_free()` — полное освобождение памяти;
- `lru_put()` — вставка или обновление элемента по ключу с автоматическим вытеснением самого старого элемента при переполнении;
- `lru_get()` — получение элемента по ключу с перемещением его в голову списка (обновление порядка использования) и увеличением счётчика Cache_hit при успешном обращении;
- `lru_size()` — получение текущего количества элементов;
- `lru_get_cache_hit()` — получение значения счётчика попаданий.

Принцип работы LRU-кеша: при вставке нового элемента он помещается в голову списка и добавляется в хеш-таблицу. При обращении к существующему элементу он перемещается в голову списка, а счётчик Cache_hit увеличивается. Если размер кеша превышает ёмкость, удаляется элемент из хвоста списка (самый старый) и соответствующая запись из хеш-таблицы. При обновлении существующего ключа счётчик Cache_hit также увеличивается, так как это считается попаданием в кеш. Такая архитектура обеспечивает выполнение всех основных операций за O(1), что делает LRU-кеш эффективным инструментом для кеширования данных в высоконагруженных системах.

В проекте реализована полная инкапсуляция LRU-кеша. Структура `struct LRUCache_t` полностью скрыта от пользователя: в заголовочном файле объявлен только непрозрачный указатель, а все внутренние поля (`ht` — указатель на хеш-таблицу, `list` — указатель на двусвязный список, `capacity` — максимальная ёмкость, `size` — текущий размер, `Cache_hit` — счётчик попаданий) и вспомогательные функции находятся в `.c` файле. Пользователь работает исключительно через публичный интерфейс. Такой подход полностью изолирует внутреннюю реализацию кеша, защищает данные от случайных изменений и позволяет модифицировать внутреннюю логику без изменения пользовательского кода.

Управление памятью в LRU-кеше построено на принципе, что пользователь самостоятельно выделяет и освобождает память для значений. Кеш освобождает только свои внутренние структуры (хеш-таблицу и список), не трогая пользовательские данные, что даёт пользователю полный контроль над временем жизни объектов и исключает двойное освобождение памяти.

## Тестирование

Для каждого модуля проекта разработан полный набор тестов, покрывающих все сценарии использования. Тесты используют динамически выделяемую память и полностью освобождают её после завершения, что позволяет проверить отсутствие утечек через Valgrind. Каждый тест выполняется независимо и выводит результат, обеспечивая наглядность и автоматизацию процесса тестирования.

### Тестирование хеш-таблицы

Для хеш-таблицы разработан набор тестов, покрывающих все сценарии использования. Тесты проверяют создание и уничтожение хеш-таблицы, вставку новых ключей и обновление существующих, поиск значений по ключу, удаление элементов из разных позиций (включая удаление из середины цепочки при коллизиях), а также корректную работу с отрицательными ключами. Отдельные тесты проверяют автоматическое рехеширование при превышении коэффициента загрузки и обработку краевых случаев, таких как создание таблицы с нулевым размером и операции с NULL-указателями.

### Тестирование двусвязного списка

Для двусвязного списка разработан набор тестов, покрывающих все основные сценарии использования. Тесты проверяют создание и уничтожение списка, добавление элементов в голову, перемещение существующих узлов в голову, удаление элементов из хвоста, удаление произвольных узлов из разных позиций, а также полную очистку списка. Отдельные тесты проверяют корректную работу со списками из одного и двух элементов, где важно правильное обновление указателей головы и хвоста. Также реализован тест для проверки работы списка с динамическими значениями (строками), что подтверждает универсальность хранения данных через void*.

### Тестирование LRU-кеша

Для LRU-кеша разработан набор тестов, покрывающих все ключевые сценарии использования. Тесты проверяют создание и уничтожение кеша, базовые операции вставки и получения элементов с проверкой корректности значений. Отдельные тесты проверяют механизм вытеснения (eviction) — при добавлении элемента в заполненный кеш удаляется самый старый элемент, а также проверяется корректность порядка использования: если к элементу обратились через `lru_get()`, он становится самым свежим и не вытесняется при следующей вставке. Реализованы тесты для работы с отрицательными ключами и хранением строковых значений (через void*), что подтверждает универсальность хранения данных. Также есть тест для кеша ёмкостью в один элемент, проверяющий, что при каждой вставке старый элемент корректно вытесняется. Отдельный тест проверяет правильность работы счётчика Cache_hit, который увеличивается при каждом успешном обращении к существующему элементу (как через `lru_get()`, так и через `lru_put()` при обновлении ключа).

## Valgrind

В процессе разработки проекта **Valgrind** использовался на протяжении всего цикла создания для каждого модуля. Проверки проводились отдельно для хеш-таблицы, двусвязного списка и LRU-кеша, а также для основной программы. Утилита Valgrind с флагами `--leak-check=full` и `--show-leak-kinds=all` позволила выявить и устранить все утечки памяти и ошибки работы с памятью на ранних этапах разработки. Благодаря этому каждый модуль проходит проверку без потери памяти, а общее количество аллокаций строго соответствует количеству освобождений. Это гарантирует стабильную и надёжную работу всего приложения в условиях длительного использования.

## Сборка и запуск

Для сборки проекта используется **Makefile**, который автоматизирует процессы компиляции и сборки всех компонентов.

### Структура Makefile

Makefile поддерживает сборку следующих целей:

- `all` — сборка основного приложения и всех тестов;
- `program` — сборка только основного приложения (`project`);
- `run_test_hashtable` — сборка тестов для хеш-таблицы;
- `run_test_list` — сборка тестов для двусвязного списка;
- `run_test_lru` — сборка тестов для LRU-кеша.

### Основные команды

| Команда | Описание |
|---------|----------|
| `make all` | Собрать основное приложение и все тесты |
| `make project` | Собрать только основное приложение |
| `make run_tests` | Собрать и запустить все тесты |
| `make test_hash` | Собрать и запустить тесты хеш-таблицы |
| `make test_list` | Собрать и запустить тесты двусвязного списка |
| `make test_lru` | Собрать и запустить тесты LRU-кеша |
| `make clean` | Очистить все собранные файлы |

### Проверка памяти (Valgrind)

Для проверки на утечки памяти предусмотрены следующие цели:

| Команда | Описание |
|---------|----------|
| `make valgrind` | Проверить основное приложение на утечки памяти |
| `make valgrind_hash` | Проверить тесты хеш-таблицы |
| `make valgrind_list` | Проверить тесты двусвязного списка |
| `make valgrind_lru` | Проверить тесты LRU-кеша |

## Пример работы проекта

Рассмотрим работу LRU-кеша на конкретном примере. Пусть ёмкость кеша равна 2, а на вход поступает последовательность из 6 запросов: `1 2 1 2 1 2`.

**Алгоритм работы:**

1. Запрос `1` — кеш пуст, страница загружается (промах)
2. Запрос `2` — страница загружается (промах), кеш заполнен
3. Запрос `1` — страница уже в кеше (попадание)
4. Запрос `2` — страница уже в кеше (попадание)
5. Запрос `1` — страница уже в кеше (попадание)
6. Запрос `2` — страница уже в кеше (попадание)

**Результат:** 4 попадания в кеш.

---

## Пример кода

```c
#include "lru_cache.h"
#include <stdio.h>
#include <stdlib.h>

int main()
{
    int capacity, n;
    scanf("%d %d", &capacity, &n);
    
    struct LRUCache_t* cache = lru_create(capacity);
    
    for (int i = 0; i < n; ++i)
    {
        int key;
        scanf("%d", &key);
        lru_put(cache, key, NULL);
    }
    
    printf("%d\n", lru_get_cache_hit(cache));
    
    lru_free(cache);
    return 0;
}
