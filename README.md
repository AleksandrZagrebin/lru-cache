# lru-cache

**LRU Cache implementation in C with hash table and doubly linked list**

---

## Кеш

**Кеш** — это временное высокоскоростное хранилище, которое ускоряет доступ к данным за счёт сохранения копий часто используемой информации. Он работает на основе принципа локальности: если данные запросили один раз, вероятно, они понадобятся снова. Кеши используются повсеместно — в процессорах, базах данных, веб-браузерах, CDN-сетях и операционных системах. Без кеширования современные системы работали бы на порядок медленнее, но из-за ограниченного размера возникает задача эффективного управления памятью.

---

## LRU-кеш

**LRU-кеш** (Least Recently Used) — это алгоритм управления кешем, который при заполнении удаляет элемент, к которому дольше всего не обращались. Он реализуется с помощью хеш-таблицы для быстрого поиска за O(1) и двусвязного списка для быстрой вставки и удаления за O(1) с отслеживанием порядка использования: голова — самый свежий элемент, хвост — самый старый. При каждом обращении элемент перемещается в голову, а при переполнении удаляется хвост, что обеспечивает постоянную скорость выполнения всех операций независимо от размера кеша. LRU широко применяется в веб-браузерах, базах данных и системах кеширования благодаря простоте и эффективности в сценариях с локальностью обращений.

## Хеш-таблица

В рамках проекта была разработана **собственная хеш-таблица**, реализующая **метод цепочек** для разрешения коллизий. Это означает, что каждый элемент массива (корзина) является указателем на связный список узлов, что позволяет хранить несколько ключей с одинаковым хешем. Для работы с хеш-таблицей реализован полный набор операций: создание таблицы с размером, округлённым до простого числа; вставка или обновление значения по ключу с автоматическим рехешированием при превышении коэффициента загрузки 0.75; поиск значения по ключу за O(1); удаление элемента по ключу; очистка всех элементов; полное освобождение памяти; отладочный вывод содержимого. Хеш-таблица спроектирована как **самостоятельный модуль** с **файловой инкапсуляцией**: в заголовочном файле объявлен только непрозрачный указатель на структуру, а все внутренние структуры и поля скрыты в `.c` файле, что обеспечивает полную изоляцию внутренней реализации и защиту данных от прямого доступа. Ключи в хеш-таблице являются **целочисленными**, что обеспечивает простоту и высокую скорость хеширования, а значения хранятся в виде **указателей на тип void**, что позволяет использовать её для хранения данных любого типа — будь то целые числа, строки, структуры или указатели на другие объекты. Хеширование ключа - остаток от деления на размер таблицы. Инкапсуляция позволяет изменять внутреннюю логику хеш-таблицы без изменения пользовательского кода, делая модуль гибким и надёжным.

### Две версии хеш-таблицы

В проекте реализованы **две версии хеш-таблицы**: одна для основного LRU-кеша, другая для тестирования. Это связано с тем, что в основном приложении хеш-таблица хранит **указатели на узлы двусвязного списка**, а управлением памятью этих узлов занимается сам список — он создаёт и освобождает узлы и их значения. Поэтому в основной версии хеш-таблица **не освобождает** значение при удалении узла, чтобы избежать двойного освобождения памяти. Для тестов же была создана отдельная версия хеш-таблицы, которая **освобождает** значение при удалении узла, поскольку в тестах хеш-таблица используется изолированно и сама управляет всей выделенной памятью. Такое разделение позволяет сохранить корректную работу LRU-кеша, где памятью управляет список, и при этом полноценно тестировать хеш-таблицу без утечек памяти. Обе версии используют одинаковый интерфейс и отличаются только внутренней логикой освобождения памяти, что демонстрирует гибкость архитектуры и грамотное разделение ответственности между модулями.

### Тестирование хеш-таблицы

Для хеш-таблицы разработан полный набор тестов, покрывающих все сценарии использования. Тесты проверяют создание и уничтожение хеш-таблицы, вставку новых ключей и обновление существующих, поиск значений по ключу, удаление элементов из разных позиций (включая удаление из середины цепочки при коллизиях), а также корректную работу с отрицательными ключами. Отдельные тесты проверяют автоматическое рехеширование при превышении коэффициента загрузки и обработку краевых случаев, таких как создание таблицы с нулевым размером и операции с NULL-указателями. Тесты используют динамически выделяемую память и полностью освобождают её после каждого теста, что позволяет проверить отсутствие утечек через Valgrind. Каждый тест выполняет проверки и выводит результат, что обеспечивает наглядность и автоматизацию процесса тестирования.

## Двусвязный список

В рамках проекта был разработан **двусвязный список**, который используется для поддержания порядка использования элементов в LRU-кеше. Список организован таким образом, что **голова** всегда содержит самый свежий (недавно использованный) элемент, а **хвост** — самый старый. Это позволяет эффективно перемещать элементы при каждом обращении и быстро удалять наименее используемый элемент при переполнении кеша.

В списке реализованы все необходимые операции: создание пустого списка; добавление элемента в голову; удаление элемента из хвоста; перемещение существующего элемента в голову (используется при cache hit); удаление произвольного элемента; очистка всех элементов; полное освобождение памяти. Структура узла содержит целочисленный ключ, а также указатель на значение типа `void*`, что позволяет хранить в списке данные любого типа: целые числа, строки, структуры или указатели на другие объекты. Кроме того, узел содержит указатели на предыдущий и следующий элементы, обеспечивающие двустороннюю навигацию. Разработанный список обеспечивает выполнение всех операций за O(1), что критически важно для производительности LRU-кеша.

### Инкапсуляция списка

В проекте реализована **частичная инкапсуляция** двусвязного списка. Структура самого списка полностью скрыта от пользователя: в заголовочном файле объявлен только непрозрачный указатель, а все внутренние поля и вспомогательные функции находятся в `.c` файле. Это защищает внутреннее состояние списка от случайных изменений и гарантирует, что пользователь работает только через публичный интерфейс. В то же время **структура узла открыта** и доступна пользователю: поля ключа, значения и указателей видны напрямую. Такое решение принято осознанно, поскольку LRU-кеш активно работает с узлами списка — читает ключи для удаления из хеш-таблицы, обновляет значения, перемещает узлы. Открытость узлов позволяет кешу получать прямой доступ к данным без дополнительных геттеров и сеттеров, что упрощает код и повышает производительность. Таким образом, достигается баланс между инкапсуляцией (защита целостности списка) и гибкостью (свободная работа с узлами), что оптимально для архитектуры LRU-кеша.

### Тестирование списка

Для двусвязного списка разработан полный набор тестов, покрывающих все основные сценарии использования. Тесты проверяют создание и уничтожение списка, добавление элементов в голову, перемещение существующих узлов в голову, удаление элементов из хвоста, удаление произвольных узлов из разных позиций, а также полную очистку списка. Отдельные тесты проверяют корректную работу со списками из одного и двух элементов, где важно правильное обновление указателей головы и хвоста. Также реализован тест для проверки работы списка с динамическими значениями (строками), что подтверждает универсальность хранения данных. Все тесты используют динамически выделяемую память и полностью освобождают её после каждого теста. Каждый тест выполняет проверки и выводит результат, что обеспечивает наглядность и автоматизацию процесса тестирования.

## LRU-кеш

В рамках проекта разработан **LRU-кеш**, который объединяет хеш-таблицу и двусвязный список для эффективного управления ограниченным объёмом памяти. Кеш состоит из трёх основных компонентов: хеш-таблицы для быстрого поиска элементов по ключу за O(1), двусвязного списка для поддержания порядка использования (голова — самые свежие элементы, хвост — самые старые), а также счётчика попаданий, который отслеживает общее количество успешных обращений к существующим элементам.

### Методы LRU-кеша

Для работы с LRU-кешем реализованы следующие операции: создание кеша с заданной ёмкостью; полное освобождение памяти; вставка или обновление элемента по ключу с автоматическим вытеснением самого старого элемента при переполнении; получение элемента по ключу с перемещением его в голову списка (обновление порядка использования) и увеличением счётчика попаданий; получение текущего количества элементов; получение значения счётчика попаданий.

При вставке нового элемента он помещается в голову списка и добавляется в хеш-таблицу. При обращении к существующему элементу он перемещается в голову списка, а счётчик попаданий увеличивается. Если размер кеша превышает ёмкость, удаляется элемент из хвоста списка (самый старый) и соответствующая запись из хеш-таблицы. Такая архитектура обеспечивает выполнение всех основных операций за O(1), что делает LRU-кеш эффективным инструментом для кеширования данных в высоконагруженных системах.

### Инкапсуляция LRU-кеша

В проекте реализована **полная инкапсуляция LRU-кеша**. Структура кеша полностью скрыта от пользователя: в заголовочном файле объявлен только непрозрачный указатель, а все внутренние поля (указатели на хеш-таблицу и список, ёмкость, текущий размер, счётчик попаданий) и вспомогательные функции находятся в `.c` файле. Пользователь работает исключительно через публичный интерфейс. Такой подход полностью изолирует внутреннюю реализацию кеша, защищает данные от случайных изменений и позволяет модифицировать внутреннюю логику (например, изменить стратегию вытеснения или добавить новую статистику) без изменения пользовательского кода. Инкапсуляция также гарантирует, что пользователь не может напрямую обратиться к хеш-таблице или списку, нарушив их целостность, что делает использование кеша безопасным и надёжным.

Важно отметить, что хеш-таблица и двусвязный список, входящие в состав кеша, также имеют собственную инкапсуляцию, что создаёт многоуровневую защиту данных: каждый модуль отвечает только за свою внутреннюю логику, а LRU-кеш выступает как высокоуровневый интегратор, связывающий их в единое целое. Такая архитектура обеспечивает чистоту интерфейсов, упрощает тестирование каждого компонента по отдельности и даёт возможность легко заменять или модифицировать отдельные части системы без влияния на остальной код.

### Тестирование LRU-кеша

Для LRU-кеша разработан полный набор тестов, покрывающих все ключевые сценарии использования. Тесты проверяют создание и уничтожение кеша, базовые операции вставки и получения элементов с проверкой корректности значений. Отдельные тесты проверяют механизм вытеснения — при добавлении элемента в заполненный кеш удаляется самый старый элемент, а также проверяется корректность порядка использования: если к элементу обратились через операцию получения, он становится самым свежим и не вытесняется при следующей вставке. Реализованы тесты для работы с отрицательными ключами и хранением строковых значений, что подтверждает универсальность хранения данных. Также есть тест для кеша ёмкостью в один элемент, проверяющий, что при каждой вставке старый элемент корректно вытесняется. Все тесты используют динамически выделяемую память и полностью освобождают её после завершения, что гарантирует отсутствие утечек, подтверждённое через Valgrind. Каждый тест выполняет проверки и выводит результат, обеспечивая наглядность и автоматизацию процесса тестирования.

В процессе разработки проекта **Valgrind** использовался на протяжении всего цикла создания для каждого модуля. Проверки проводились отдельно для хеш-таблицы, двусвязного списка и LRU-кеша, а также для основной программы. Утилита Valgrind с флагами полной проверки утечек позволила выявить и устранить все ошибки работы с памятью на ранних этапах разработки. Благодаря этому каждый модуль проходит проверку без потери памяти, а общее количество аллокаций строго соответствует количеству освобождений. Это гарантирует стабильную и надёжную работу всего приложения в условиях длительного использования.

## Пример использования

Рассмотрим работу LRU-кеша на конкретном примере. Пусть ёмкость кеша равна 2, а на вход поступает последовательность из 6 запросов: `1 2 1 2 1 2`.

**Алгоритм работы:**

1. Запрос `1` — кеш пуст, страница загружается (промах)
2. Запрос `2` — страница загружается (промах), кеш заполнен
3. Запрос `1` — страница уже в кеше (попадание)
4. Запрос `2` — страница уже в кеше (попадание)
5. Запрос `1` — страница уже в кеше (попадание)
6. Запрос `2` — страница уже в кеше (попадание)

**Результат:** 4 попадания в кеш.

## Пример кода

```c
#include "lru_cache.h"
#include <stdio.h>
#include <stdlib.h>

int main()
{
    int capacity, n;
    scanf("%d %d", &capacity, &n);
    
    struct LRUCache_t* cache = lru_create(capacity);
    
    for (int i = 0; i < n; ++i)
    {
        int key;
        scanf("%d", &key);
        lru_put(cache, key, NULL);
    }
    
    printf("%d\n", lru_get_cache_hit(cache));
    
    lru_free(cache);
    return 0;
}
