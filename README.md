# lru-cache
LRU Cache implementation in C with hash table and doubly linked list
## Кеш

**Кеш** — это временное высокоскоростное хранилище, которое ускоряет доступ к данным за счёт сохранения копий часто используемой информации. Он работает на основе принципа локальности: если данные запросили один раз, вероятно, они понадобятся снова. Кеши используются повсеместно — в процессорах, базах данных, веб-браузерах, CDN-сетях и операционных системах. Без кеширования современные системы работали бы на порядок медленнее, но из-за ограниченного размера возникает задача эффективного управления памятью.

## LRU-кеш

**LRU-кеш** (Least Recently Used) — это алгоритм управления кешем, который при заполнении удаляет элемент, к которому дольше всего не обращались. Он реализуется с помощью хеш-таблицы для быстрого поиска за O(1) и двусвязного списка для быстрой вставки и удаления за O(1) с отслеживанием порядка использования: голова — самый свежий элемент, хвост — самый старый. При каждом обращении элемент перемещается в голову, а при переполнении удаляется хвост, что обеспечивает постоянную скорость выполнения всех операций независимо от размера кеша. LRU широко применяется в веб-браузерах, базах данных и системах кеширования благодаря простоте и эффективности в сценариях с локальностью обращений.

## Хеш-таблица

В рамках проекта была разработана **собственная хеш-таблица**, реализующая **метод цепочек** для разрешения коллизий. Это означает, что каждый элемент массива (корзина) является указателем на связный список узлов, что позволяет хранить несколько ключей с одинаковым хешем. Для работы с хеш-таблицей реализованы ключевые методы: `hash_table_create()` — создание таблицы с размером, округлённым до простого числа; `hash_table_put()` — вставка или обновление значения по ключу с автоматическим рехешированием при превышении коэффициента загрузки 0.75; `hash_table_get()` — поиск значения по ключу за O(1); `hash_table_remove()` — удаление элемента по ключу; `hash_table_clear()` — очистка всех элементов; `hash_table_free()` — полное освобождение памяти; `hash_table_print()` — отладочный вывод содержимого. Хеш-таблица спроектирована как **самостоятельный модуль** с **файловой инкапсуляцией**: в заголовочном файле объявлен только непрозрачный указатель на структуру `struct HashTable_t`, а все внутренние структуры (`HashNode_t`, поля `buckets`, `size`, `count`, `load_factor`) и вспомогательные функции скрыты в `.c` файле, что обеспечивает полную изоляцию внутренней реализации и защиту данных от прямого доступа. Ключи в хеш-таблице являются **целочисленными (`int`)**, что обеспечивает простоту и высокую скорость хеширования, а значения хранятся в виде **указателей на тип void (`void*`)**, что позволяет использовать её для хранения данных любого типа — будь то целые числа, строки, структуры или указатели на другие объекты. Такая инкапсуляция позволяет изменять внутреннюю логику хеш-таблицы без изменения пользовательского кода, делая модуль гибким и надёжным.

### Две версии хеш-таблицы

В проекте реализованы **две версии хеш-таблицы**: одна для основного LRU-кеша, другая для тестирования. Это связано с тем, что в основном приложении хеш-таблица хранит **указатели на узлы двусвязного списка** (`struct ListNode_t*`), а управлением памятью этих узлов занимается сам список — он создаёт и освобождает узлы и их значения. Поэтому в основной версии хеш-таблица **не освобождает** значение (`value`) при удалении узла, чтобы избежать двойного освобождения памяти. Для тестов же была создана отдельная версия хеш-таблицы (файл `hashtable_test.c`), которая **освобождает** значение при удалении узла, поскольку в тестах хеш-таблица используется изолированно и сама управляет всей выделенной памятью. Такое разделение позволяет сохранить корректную работу LRU-кеша, где памятью управляет список, и при этом полноценно тестировать хеш-таблицу без утечек памяти. Обе версии используют одинаковый интерфейс и отличаются только внутренней логикой освобождения памяти, что демонстрирует гибкость архитектуры и грамотное разделение ответственности между модулями.

### Тестирование хеш-таблицы

Для хеш-таблицы разработан полный набор тестов, покрывающих все сценарии использования. Тесты проверяют создание и уничтожение хеш-таблицы, вставку новых ключей и обновление существующих, поиск значений по ключу, удаление элементов из разных позиций (включая удаление из середины цепочки при коллизиях), а также корректную работу с отрицательными ключами. Отдельные тесты проверяют автоматическое рехеширование при превышении коэффициента загрузки и обработку краевых случаев, таких как создание таблицы с нулевым размером и операции с NULL-указателями. Тесты используют динамически выделяемую память и полностью освобождают её после каждого теста, что позволяет проверить отсутствие утечек через Valgrind. Каждый тест выполняет проверки и выводит результат, что обеспечивает наглядность и автоматизацию процесса тестирования.
